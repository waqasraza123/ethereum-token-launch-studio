# Phase 2 core schema

## Goal

Create the first production-grade business schema in Supabase before touching auth UI, admin forms, or project dashboards.

## What this step establishes

- a replayable migration chain through `0002_phase_2_core_business_schema.sql`
- `app_public.workspaces`
- `app_public.workspace_members`
- `app_public.projects`
- stable `uuid` primary keys
- validated lowercase slug rules
- foreign keys between workspaces, members, and projects
- a future-facing foreign key from workspace members to `auth.users`
- reusable `updated_at` trigger behavior
- migration replay validation in CI

## Assumptions

- IDs are written explicitly by the future app/server layer rather than generated by database defaults in this step
- one user may belong to many workspaces
- project slugs are unique per workspace, not globally
- multiple owners are allowed unless a later product rule forbids it

## What is intentionally deferred

- auth wiring
- invitations
- protected routes
- RLS policies
- project settings beyond name, slug, and description
- contract registry tables
- activity/event tables
- storage buckets and file records

## Acceptance criteria for this step

- `pnpm db:list` shows both migrations in order
- `pnpm db:validate` passes
- `pnpm db:replay:check` passes from zero
- migration tests prove schemas, tables, triggers, and key constraints
- full repo verification remains green

## Risks that remain

- auth is not wired yet
- no RLS exists yet
- no workspace creation flow exists yet
- no admin UI reads or writes this schema yet
- contract deployment persistence is still deferred
